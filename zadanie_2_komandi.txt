23 Пользователи без отзывов

Подзапрос
User.where.not(id: Review.select(:user_id))

left_outer_joins
User.left_joins(:reviews).where(reviews: { id: nil })

Подзапрос - проще читается, рельсы делают not in (select user_id from reviews). 
Проблема в том, что если хоть один user_id в reviews будет null, 
то select user_id from reviews - вернёт пустой результат (особенность sql).
left join - присоединяет reviews к каждому user, оставляет только тех, 
у кого нет совпадений (reviews.id is null). безопаснее при возможных null, 
обычно эффективнее на больших таблицах.

Ответы на вопросы:
1. Индексы:
В текущей системе сделаны индексы:
Пользователь по емейлу, Пользователь как владелец события, Айди категории события,
Айди места проведения события, Индекс по дате события, Билеты по отношению к определенному событию,
Билеты по отношению к определенному пользователю, Билеты по статусу, Место проведения мероприятия по городам,
Айди спонсора(уникальный), Визит события(уникальный), Статус отзывов, Прочитанность уведомлений

Ну можно спонсорам еще добавить индекс для поиска по айди, например - но вроде как это не большая таблица будет



Разница includes, preload, joins, references, eager_load

includes — универсальный, рейлс сам решает делать 2 запроса или JOIN.
Если в where нет условий по связи, то 2 запроса. Если в where есть условие по связанной таблице, то автоматически JOIN.
Event.includes(:category)
Event.includes(:category).where(categories: { title: 'Music' }) - тут автоматически JOIN применит

preload — делает всегда два отдельных запроса.


eager_load — всегда left_outer_join в одном запросе, может фильтровать.

joins — делает inner join


references — явно указывает includes что нужен join, когда рельса не может угадать сама что использовать


2. Проблема N+1

N+1 — это когда для каждой записи из первого запроса выполняется ещё один запрос.
Нужно этого избегать, например, чтобы понять сколько событий в категории можно написать код двумя способами:
если написать: events = Event.all
events.each do |e|
  puts e.category.title
end
 - то при 5 событиях - такой запрос сделает 6 запросов, а если использовать
Event.includes(:category)
- то будет 2 запроса, и читается лучше


 3. Типы связей в Rails 

belongs_to — FK у меня. Данный объект принадлежит другому, например,
 в таблице tickets есть event_id

has_one — FK в другой таблице. Связь один к одному, 
например один пользователь имеет один аккаунт.

has_many — FK у другого. Связь один имеет многих. Например у события много билетов

has_many :through — многие ко многим через join модель, + могут быть доп данные. 
Например, как у многих спонсоров есть разные ивенты, так и у ивентов есть разные спонсоры.
Связаны они между собой с помощью третьей таблице, в которой есть айди строки, 
айди ивента, айди спонсора и другая информация, 
например, сколько спонсор денег внес на событие

has_one :through — один к одному через промежуточную модель, с примером что-то трудно, 
не придумаю и у себя в текущем приложении не использовал

has_and_belongs_to_many (HABTM) — многие ко многим без модели join таблицы.
Нет доп. данных, использовать только для очень простых связей, 
там будет таблица из двух колонок: айди первой и второй сущностей.


 4. Для чего нужны индексы 

Прежде всего для того, чтобы ускорять поиск по таблице, перебор данных таблицы. 
При индексах таблица разделяется на б-дерево. Б-дерево состоит из узлов, каждому узлу 
задано значение. Так, когда sql проходится по таблице - он не перебирает строки по порядку, 
а сравнивает значение данных для поиска со значением текущего узла, если значение больше -
то идет в правый узел, если меньше, то в левый и тд.
Но индексы замедляют изменение таблиц, так как при изменении таблицы - постгресу необходимо
переиндексировать таблицу. 
Так что для больших таблиц индексы нужны, для маленьких - точно нет.


 5. Разница delete и destroy 

destroy — вызывает коллбеки
delete — не вызывает коллбеки


 6. Опасность default_scope 

default_scope автоматически добавляет условие/фунционал ко всем запросам модели. 
Это плохо, так как даже если это и не будет мешать при дальнейшей работе - то все равно
можно про этот default_scope забыть и накосячить, либо новый разработчик вообще о нем может не знать
